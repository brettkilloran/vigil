# Review Audit — Import Flow

Scope: `/review-light` on the import flow (`lore-import` pipeline, related API routes, and import docs).

## CRITICAL

- None.

## HIGH

### 1) Direct `/plan` fallback can produce plans that cannot be applied
- Why it matters: The UI fallback can show a valid plan even when no `lore_import_jobs` row exists, but apply requires server job metadata and throws; this breaks end-to-end import completion for fallback users.
- Evidence: `391:405:src/lib/lore-import-apply.ts`
- Fix direction: Ensure fallback planning creates/binds a server job row for the same `importBatchId`, or prevent apply when a job-backed batch is not present.

### 2) Full-table `spaces` scan in plan build hot path
- Why it matters: Import planning loads all spaces every time to build path labels, creating unbounded work and poor scaling under concurrent imports.
- Evidence: `301:307:src/lib/lore-import-plan-build.ts`
- Fix direction: Query only required ids (candidate spaces + ancestor chain) or cache a bounded space map for the operation.

### 3) Progress-event writes re-aggregate JSON array on each event
- Why it matters: Rebuilding bounded JSON arrays in-row per event amplifies write cost and lock contention on active jobs.
- Evidence: `117:133:src/lib/lore-import-job-process.ts`
- Fix direction: Batch/coalesce event writes or move events to append-only rows to reduce per-event update cost.

### 4) API docs do not document divergent import text limits
- Why it matters: `/api/lore/import/plan` and `/api/lore/import/jobs` accept different max text sizes, but canonical API docs do not state this, causing contract confusion and avoidable failures.
- Evidence: `20:27:app/api/lore/import/plan/route.ts`
- Fix direction: Document both limits and when to choose sync plan vs async jobs in `docs/API.md`.

## MEDIUM

### 5) Client-provided `importBatchId` can replace another batch's pending review rows
- Why it matters: `persistImportReviewRows` deletes pending items by `(spaceId, importBatchId)`; accepting arbitrary client batch ids enables accidental or malicious clobber within the same space.
- Evidence: `20:27:app/api/lore/import/plan/route.ts`
- Fix direction: Require server-issued batch ids (or proof-of-ownership token/job binding) before replacing pending review rows.

### 6) Invalid multipart `context` is silently ignored in parse route
- Why it matters: Non-empty but malformed context can quietly drop user intent, leading to unexpected planning scope without clear feedback.
- Evidence: `288:299:app/api/lore/import/parse/route.ts`
- Fix direction: Return a structured warning/error when provided `context` fails parse/schema validation.

### 7) Kind-mapping docs omit `lore_source` persisted entity type
- Why it matters: The matrix appears complete but misses the source-document type, making future extension/search logic easier to implement incorrectly.
- Evidence: `15:25:docs/LORE_IMPORT_KIND_MAPPING.md`
- Fix direction: Add a row/footnote for source document cards mapping to `lore_source`.

## LOW

### 8) Import docs imply stricter `entity_type` closure than general item create API enforces
- Why it matters: Import pipeline uses registry-backed entity types, but generic item creation still accepts open string values; this mismatch can mislead maintainers about global invariants.
- Evidence: `36:65:app/api/spaces/[spaceId]/items/route.ts`
- Fix direction: Explicitly document this boundary, and consider API-side narrowing only if product intends global closure.

## Residual Risk

- Legacy import endpoints (when enabled) remain less constrained than the primary jobs/apply path.
- `gm_workspace` planning still loads a full space map for path labels; subtree scope uses the bounded `loadSpaceMapForLoreImportPathLabels` path only.
