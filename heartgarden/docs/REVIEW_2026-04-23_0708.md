The structured-document feature is in strong shape overall: build/test/storybook gates pass, the new shared body contract is coherent, and heading-aware retrieval landed cleanly with fallback for headingless documents. The biggest remaining risk is at the MCP boundary, where markdown/blocks inputs do not yet share the same fail-fast size guard as `content_text`, so runaway generated payloads can still overrun request budgets. Aside from that, the remaining issues are low-friction hardening and docs hygiene. None of the findings suggest a release blocker for local usage, but one contract-level decision is still needed for MCP safety posture. The section-aware indexing and citation threading look directionally correct and worth preserving.

## MEDIUM

### MCP structured payload size cap parity

- Severity + classifier: **MEDIUM · RISKY**
- confidence: **0.78**
- User-facing impact: Very large AI-generated markdown/blocks sent via MCP can bypass the 2M fail-fast guard and cause slow or failed create/patch requests.
- Evidence:

```1160:1176:src/lib/mcp/heartgarden-mcp-server.ts
      if (typeof args.content_text === "string") {
        const tooLong = mcpContentTextTooLong("content_text", args.content_text);
        if (tooLong) {
          return { content: [{ type: "text", text: tooLong }], isError: true };
        }
      }
      const hasExplicitJson =
        args.content_json != null &&
        typeof args.content_json === "object";
      const markdownInput = typeof args.content_markdown === "string" ? args.content_markdown : "";
      const hasMarkdown = markdownInput.trim().length > 0;
      const hasBlocks = Array.isArray(args.content_blocks);
```

- Fix direction: Apply the same max-length guard to `content_markdown` and (serialized) `content_json`/`content_blocks` before conversion, and keep the tool contract explicit about enforced limits.
