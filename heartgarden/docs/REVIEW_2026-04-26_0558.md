# Review Audit — 2026-04-26 0558 UTC

## Plain Summary

Three `compositor-2-fast` passes reviewed five CSS changes to `ArchitecturalCanvasApp.module.css` made to fix `backdrop-filter` blur on toolbar glass panels and the boot screen dock. **No security or data-integrity issues were found** — the changes are purely visual/rendering. The root-cause fix (removing idle `will-change: filter` from `.viewport` and `.chromeLayer`) is correct: those properties were creating compositing isolation that silently broke `backdrop-filter` sampling in all child and sibling panels. One stale comment was left behind that contradicts the current code and could mislead a future maintainer; it is auto-fixed. One theoretical performance concern exists (a possible one-frame hitch on low-end GPUs when the focus/gallery blur activates) but its confidence is too low to act on without profiling data — noted here for awareness. Everything else is clean.

### L2 — Filter transition without pre-promotion hint: theoretical jank on weak GPUs
- **Severity:** LOW · `RISKY`
- **Confidence:** 0.50
- **User-facing impact:** When focus mode or the gallery opens, there is a slim chance of a one-frame stutter on very weak integrated GPUs as the browser promotes `.viewport` to a compositing layer on-the-fly rather than at idle. The 0.28s eased transition distributes cost across many frames, which greatly reduces the likelihood.
- **Evidence:** `42:48:heartgarden/src/components/foundation/ArchitecturalCanvasApp.module.css`, `106:119:heartgarden/src/components/foundation/ArchitecturalCanvasApp.module.css`
- **Fix direction (if profiling confirms):** Rather than restoring idle `will-change: filter` on `.viewport` (which breaks `backdrop-filter` sampling), scope the blur to a smaller subtree or add a brief `requestAnimationFrame` delay before applying `.shellBackdropBlurActive` so the browser can compose on idle. Do not act without a real profiling trace showing a dropped frame.
- **Terminal state:** `QUESTION_FOR_USER` (see Questions Phase)

## What is Working Well (Don't Break These)

- **`backdrop-filter` on all toolbar glass panels** now correctly samples canvas content at idle — the root-cause fix is sound and well-commented.
- **`shellBackdropBlurActive` state** still correctly blurs `.viewport` and `.chromeLayer` when focus/gallery is open, using `filter: blur(12px) saturate(90%)` with `will-change: filter` scoped only to that active state.
- **Boot screen dock panel** (`VigilAppBootScreen.bootChromeDockPanel`) is unblocked — it no longer fights a compositing layer from `.viewport`.
- **`prefers-reduced-motion` block** is clean: `transition: none` applies to both `.viewport` and `.chromeLayer`, and the removed `will-change: auto` reset was redundant once the base rule was cleaned up.
- **Button selectors are correct**: `shellTopLogOutTrigger` and `statusSaveBarTrigger` match the actual DOM class usage in `ArchitecturalCanvasApp.tsx` and `ArchitecturalStatusBar.tsx`. Hover/active states still work — global `vigil-btn[data-tone="glass"]:hover` rules carry higher specificity and override the transparent base.
- **Minimap and boot screen** do not have the same parent `will-change: filter` isolation pattern; their `backdrop-filter` glass roots use `will-change: backdrop-filter` on the element itself, which is the correct pattern.
