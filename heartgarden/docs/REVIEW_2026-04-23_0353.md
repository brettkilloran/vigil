# Review: node object types and fields (light)

Scope: in-memory and API shapes for canvas nodes — `NodeTheme` / `LoreCardKind` / `CanvasEntity`, `GraphNode`, `CanvasItem`, `PERSISTED_ENTITY_TYPES`, and related create/parse paths.

## CRITICAL

None. No hard security boundary failure found in this layer; main risks are contract drift and data-shape integrity.

## HIGH

### Entity type casing is split-brain (validation vs persistence vs lore shell)

`validateItemWriteJsonPayload` lowercases `entityType` when selecting an `entityMeta` schema, but the items route kept the client string as-is for storage and for `isLoreCardPersistedEntityType`, which only matches lower-case literals. A client can send `entityType: "Character"`, pass character-shaped `entityMeta` validation, and still miss lore-shell gates or store inconsistent casing.

- **Why it matters:** Lore-card synthesis and shell checks can disagree with what validation implied; DB rows may not match the normalized form used in schema lookup.
- **Evidence:**

### `PERSISTED_ENTITY_TYPES` is a closed write set in code; HTTP create does not enforce it

The registry documents an explicit closed array for `items.entity_type`, while `POST` accepts any `entityType` string up to 64 characters. That weakens the single-source-of-truth story versus import/apply and allows arbitrary tags in the same column.

- **Why it matters:** Data integrity and predictable indexing/search behavior; harder to reason about which writers are allowed.
- **Evidence:**

```50:77:heartgarden/src/lib/lore-object-registry.ts
/**

export const PERSISTED_ENTITY_TYPES = [
  "character",
```


- **Fix direction:** Reject unknown values on create with 400, or document that this route intentionally allows a superset (and adjust registry wording). Treat as **RISKY** if any client relies on ad-hoc `entity_type` strings.

## MEDIUM

### `lore_variant` on create does not cover all `LoreCardVariant` values; v11 is character-skewed in the schema

`LoreCardVariant` includes `v4` and `v7` in the foundation model; the create body only allows `v1` | `v2` | `v3` | `v11`. `resolveLoreCardForCreate` forces character to v11, faction to v4, and only uses `loreVariant` for location (v1–v3) before defaulting.

- **Why it matters:** API consumers may think `lore_variant: "v11"` applies beyond character; for location, v7 is only reached via defaults.
- **Evidence:**

```9:12:heartgarden/src/components/foundation/architectural-types.ts
export type LoreCardKind = "character" | "faction" | "location";
/** Faction: v4 Archive-091 (canonical). Location: v2–v3 (legacy), v7 ORDO slab. Character: v11 only. */
export type LoreCardVariant = "v1" | "v2" | "v3" | "v4" | "v7" | "v11";
```


```18:33:heartgarden/src/lib/lore-item-create-synthesis.ts
export function resolveLoreCardForCreate(args: {
  kind: LoreCardKind;
  loreVariant?: LoreCardVariant | undefined;
}): LoreCard {
  const { kind } = args;
  if (kind === "character") {
    return { kind: "character", variant: "v11" };
  }
  if (kind === "faction") {
    return { kind: "faction", variant: "v4" };
  }
  const v = args.loreVariant;
  if (v === "v1" || v === "v2" || v === "v3") {
    return { kind, variant: v };
  }
  return { kind, variant: defaultLoreCardVariantForKind(kind) };
}
```

- **Fix direction:** Extend the HTTP enum to match supported explicit variants, narrow by `entityType`, or document that `lore_variant` is partial and v4/v7 are implicit.

## LOW

### `parseLoreCard` normalizes legacy variants

Character variants collapse to v11; location/faction map to a small set. Stored `hgArch.loreCard` is not a byte-for-byte round-trip for every historical variant.

- **Why it matters:** Only if migration/debug fidelity requires exact preserved variants.
- **Evidence:**

```316:337:heartgarden/src/lib/lore-node-seed-html.ts
export function parseLoreCard(raw: unknown): LoreCard | undefined {
  if (!raw || typeof raw !== "object") return;
  const o = raw as Record<string, unknown>;
  const k = o.kind;
  const v = o.variant;
  if (!isLoreCardKind(String(k))) return;
  if (typeof v !== "string") return;
  const kind = k as LoreCardKind;
  if (kind === "character") {
    /* Legacy stored variants (v3 passport, v8–v10, etc.) normalize to the canonical character template. */
    if (!v.length) return;
    return { kind: "character", variant: "v11" };
  }
  if (kind === "location") {
    if (v === "v1") return { kind: "location", variant: "v2" };
    if (v === "v2" || v === "v3" || v === "v7") return { kind: "location", variant: v };
    return;
  }
  if (kind === "faction") {
    if (v === "v1" || v === "v2" || v === "v3" || v === "v4") return { kind: "faction", variant: "v4" };
    return;
  }
  return;
}
```

- **Fix direction:** Document as intentional normalization, or add a separate legacy field if fidelity is required.
