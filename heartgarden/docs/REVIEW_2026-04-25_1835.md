This is a scaling-targeted audit asking the question: when this app holds thousands of notes, tens of thousands of entities, and hundreds of folders/spaces, where does it break? Six specialist passes — database/queries, search recall, sync/bandwidth, canvas rendering, vault index pipeline, and lore import — agreed on three structural cliffs. The biggest is that opening any workspace today fetches the **entire** subtree of items in one full-row JSON blob, with no pagination or projection; at 5–10k items the bootstrap response alone will dominate first-paint and balloon server memory. The second is the canvas connection-layout loop, which iterates every connection in the brane on every animation frame — at a few thousand edges this freezes the page even when most are off-screen. The third is the AI reindex pipeline: every write schedules an unbounded `after()` job per item with no per-item dedupe, so a 500-note import fires 500 simultaneous OpenAI/Anthropic jobs, deletes-and-reinserts every embedding row, and burns spend on a `content_hash` skip path that exists in the schema but is dead code. Search recall has parallel scaling debt: fuzzy search runs `similarity()` over `content_text` and `search_blob` with no trigram index on either column, the suggest tokenizer drops Unicode/CJK/punctuation outright, and hybrid retrieval's vector and mention legs ignore non-space filters so users will see results that violate their requested type/kind/historical filters. Foundations are mostly healthy: the prior audit (`REVIEW_2026-04-25_1730.md`) already capped `/api/graph/brane`, fixed cross-brane invariants, and added the `spaces(brane_id)` and `item_links(target_item_id)` indexes; viewport culling exists; per-space link-revision has a 1s cache; embedding batches at 64; reindex space concurrency is bounded.

## What is working well (don't break these)

- Viewport culling already exists for entities, stacks, and connections (`canvas-viewport-cull.ts`); off-canvas content does not mount. The CRITICAL canvas finding is about per-frame work over *total* edges, not DOM count.
- `reindexSpaceVault` already enforces bounded concurrency (`HEARTGARDEN_REINDEX_CONCURRENCY`, default 4) — good. Apply the same pattern to the per-PATCH `after()` queue.
- `apiPatchItem` serializes per-item PATCHes; delta-sync already defers polling while a write is in flight.
- The 1s `itemLinksRevision` cache is a sound thundering-herd damper; the structural fix (write-time counter) replaces the underlying scan, not the cache.
- The 60-second `SUBTREE_ID_SNAPSHOT_INTERVAL_MS` already cuts the worst case of `includeItemIds=1` cost — the remaining issue is a structural one (tombstone log).
- Embedding batches at 64 inputs per OpenAI call; that's a healthy floor — the fix is retry + dedupe, not batch size.
- `heading_path` is correctly maintained on chunks and consumed by `vault-retrieval.ts` for section-aware citations.
- `entity_mentions` table-level invariants are strong (unique key + CHECK + indexes from prior audit); incremental rescan + batched target-space lookup are sound.
- `/api/graph/brane` is now neighborhood-first with ETag from the prior audit — that pattern (seed + maxDepth + limit + `If-None-Match`) is the model the bootstrap split should follow.
- Lore-import job claim is `(jobId)` row-locked via `UPDATE … WHERE status IN ('queued', …)`; the missing piece is per-brane serialization, not per-job.
