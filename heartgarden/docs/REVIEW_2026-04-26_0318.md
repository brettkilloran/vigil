Reviewed the current uncommitted graph cache work with three focused fast passes. The biggest risk is a revision-consistency bug: the API can compute a fresh `itemLinksRevision` but still return a cached `graphRevision`, which can point to the wrong persisted layout for a short window. A second risk is hot-path cost during interactive drag because full layout serialization happens repeatedly before debounce settles. API/docs contract drift is moderate risk for future maintainers and external consumers. The schema, payload limits, and guarded write path are strong and should not be broken while fixing the issues below.

## HIGH

### Stale graph fingerprint can mismatch fresh links revision
- **Severity + classifier:** HIGH · RISKY
- **confidence:** 0.89
- **User-facing impact:** Users can intermittently warm-start from stale coordinates even when fresh nodes/edges were returned.
- **Evidence:** `52:57:app/api/spaces/[spaceId]/graph/route.ts`, `49:59:src/lib/space-graph-revision.ts`
- **Fix direction:** Ensure `computeSpaceGraphRevisionForSpace` does not return a stale cached value when a fresh `itemLinksRevision` is supplied; either recompute on that call path or validate cache compatibility before returning.

## MEDIUM

### Overlay drag path repeatedly serializes full layout map
- **Severity + classifier:** MEDIUM · RISKY
- **confidence:** 0.84
- **User-facing impact:** Large graphs can feel laggy during/after drag because full position serialization is done repeatedly on interactive updates.
- **Evidence:** `58:95:src/components/product-ui/canvas/LinkGraphOverlay.tsx`
- **Fix direction:** Move heavy serialization out of per-change effect path by buffering/ref-based sampling and persisting only on stable checkpoints.

### API contract drift between route responses and shared type/docs
- **Severity + classifier:** MEDIUM · RISKY
- **confidence:** 0.80
- **User-facing impact:** Integrations and future edits can silently break because the runtime response includes fields not clearly documented as part of the contract.
- **Evidence:** `175:188:app/api/spaces/[spaceId]/graph/route.ts`, `29:40:src/lib/graph-types.ts`
- **Fix direction:** Keep response typing and API documentation aligned for graph cache fields and pagination metadata.

## LOW

### Cache read path trusts stored positions without sanitizing
- **Severity + classifier:** LOW · SAFE
- **confidence:** 0.72
- **User-facing impact:** If corrupted cache rows appear, clients may render bad coordinates instead of gracefully treating cache as a miss.
- **Evidence:** `22:50:src/lib/space-graph-layout-cache.ts`
- **Fix direction:** Sanitize positions on read and return `null` cache hit when invalid.

## What is working well (don't break these)
- Payload guardrails are strong (`max bytes`, `max nodes`, shape validation) in `graph-layout-cache` write path.
- Migration/schema alignment for `space_graph_layout_cache` is coherent, including FK + unique composite key.
- Cache attachment is gated off paginated graph fetches, avoiding partial-page coordinate misuse.
- Test coverage was added for layout cache contract and synthetic revision stability helpers.
